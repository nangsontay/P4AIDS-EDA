1.2 Decoding "Hidden" Missing Values (The Iceberg Effect)

Typically, basic reports use .isnull() and conclude the data is "100% clean." However, a deep dive into the value distribution exposed a large volume of Missing Data disguised under the unknown label and the value -1.

Categorical Variables:

Variable Description Unknown Rate Severity

job Occupation 0.64% Low

Numerical Variable:

pdays (Days since last contact from previous campaign): As many as 81.74% of customers have a value of -1, indicating they were never contacted.

Insight

The poutcome unknown rate and pdays = -1 rate are nearly identical (~81.7%). This is logically sound: more than 81% of this dataset consists of entirely new customers who have never been contacted by the bank in previous campaigns.

Key Action

Absolutely do not drop rows containing unknown — this would lose >80% of data. We will keep unknown as an independent category, since "the bank has no data on this customer" is itself an important signal for prediction. For pdays, we will create an additional binary variable: is_previously_contacted (0 or 1).

1.3 Target Distribution Shock (Severe Class Imbalance)

88.30%
"No" — Declined (39,922)
11.70%
"Yes" — Agreed (5,289)

Analysis of the target variable y (whether customers opened a savings account) reveals an enormous disparity. Only 11.70% (5,289 people) agreed, while 88.30% (39,922 people) declined. The Pie and Bar charts clearly illustrate the overwhelming dominance of the "no" group.

Technical Issue

If we use Accuracy (overall correctness), a "blind" model that simply predicts all customers as "no" would achieve an Accuracy of 88.3%. This number is deceptively impressive but completely without practical value.

Committed Solution

Models will not be evaluated using Accuracy. The mandatory replacement metrics are F1-Score, Precision, Recall, and the area under the ROC-AUC curve. Additionally, the training process must apply class weighting techniques (class_weights) or synthetic data generation (SMOTE) to force the model to learn to identify the minority class ("yes").

1.4 Data Leakage Discovery — The duration Variable

This is a critical finding for ensuring the model can be deployed in practice (Deployable).

Statistical Analysis & Boxplot:

221s
"No" Group — Avg Duration
537s
"Yes" Group — Avg Duration

The data reveals an extraordinary difference in call duration (duration) between the two customer groups. The Boxplot shows the "yes" group distribution is significantly higher, with clearly breakaway interquartile ranges.

Future Leakage Risk

1. If duration = 0 (call not answered), it is certain that y = no.
2. Longer call duration proves customer interest, leading to higher closing rate.
3. HOWEVER, the duration variable is only collected after the call has ended. The bank's goal is to filter the customer list before the operator picks up the phone.

Mandatory Action

The duration variable is a dangerous form of Data Leakage. It can be used for behavioral analysis reports (e.g., "keep customers on the line for at least 500 seconds"), but it is mandatory to remove (drop) the duration column before feeding data into machine learning model training. If not, the model will be "deluded" by high performance in the test environment but fail completely in real-world deployment.

Chapter 2 Numerical Variable Analysis, Outliers & Customer "Fatigue"

2.1 Age & Balance Distribution: Solving the Outlier Problem

Age Variable:

Distributed in a right-skewed pattern, with the highest concentration among working-age customers between 30–40 years old.

Valuable Insight

From the KDE curve, age 60+ shows the "Yes" group (blue) rising higher and denser than the "No" group (red). Combined with the Boxplot whisker extending further upward: Retired customers, though a small proportion, have an extremely impressive conversion rate. They have idle cash flow and a much higher need for safe, interest-bearing savings accounts.

Balance Variable (Account Balance):

  • Contains indebted customers (negative values) and extremely wealthy individuals with balances up to nearly 100,000 EUR (visible as numerous black dots on the Boxplot).
  • The distribution is extremely sharp and narrow near zero, causing the Boxplot to appear nearly squeezed flat.
Key Action — Preprocessing

Because balance contains countless Extreme Outliers and includes negative values, traditional scaling techniques such as MinMaxScaler or StandardScaler will fail miserably. We will use RobustScaler (based on IQR interquartile range) to scale numerical variables, or use the Yeo-Johnson transformation to handle skewness while preserving the negative values of indebted customers.

Statistical noise in the tail: The "jumps" at call number 11 or 17 on the red line chart are just Statistical Noise — the number of customers called that many times is too small to be statistically meaningful.

Business Recommendation

Feature Engineering

The campaign variable extends to 50–60 calls, creating a long-tail full of noise. We will apply Binning (Grouping), converting this numerical column into categorical form: '1 call', '2–3 calls', '4–5 calls', and 'More than 5 calls' to help the model learn patterns more easily and prevent overfitting.

3.5 Preprocessing Strategy (Feature Engineering / Encoding)

Based on comprehensive analysis, the data preparation strategy for ML models:

1

Ordinal Encoding

Reserved exclusively for the education variable (primary → secondary → tertiary). Labels 0, 1, 2 are assigned to preserve the linear nature. The unknown group will be assigned the Mode value or interpolated prediction.

2

Binary Mapping

Yes/No variables such as housing, loan, default will be directly mapped to 1 and 0.

3

One-Hot Encoding (OHE)

Applied to Nominal variables with no hierarchical ordering such as marital, contact, poutcome.

4

Target Encoding / Binning

For the job variable (12 groups) and month (12 months), using OHE would cause column count explosion (Curse of Dimensionality). We will use Target Encoding or group month into Quarters (Q1–Q4) for cleaner model learning.

Chapter 4 The Core Blind Spot — Empirical Evidence of "Data Leakage"

4.1 The Logic of Data Leakage

From a business logic perspective, the bank builds an AI model to filter the customer list BEFORE the call center agent picks up the phone, to save telecommunications and staffing costs.

Critical Problem

However, duration is information only generated AFTER the call has taken place and ended. If duration = 0 (no answer), Target y = no is certain. Feeding duration into training is like giving students the answer key before the exam.

4.2 Evidence 1: Feature Importance Distortion

The comparative chart of the Top 10 Features of the 2 models reveals the "illusion" created by Data Leakage:

Model A — The Deceiver

Retaining duration

The duration variable rises to claim sole dominance with an overwhelming Importance Score (> 0.35). It acts like a "black hole," absorbing all importance from other variables. The model becomes "lazy," not learning deeper patterns about finance or demographics.

Model B — The Realist

Removing duration

When the "answer key is taken away," the model learns true patterns. Variables such as balance, age, day, and campaign rise to the top. This is the true portrait of the customer.

4.4 Key Action — Data Scientist's Decision

Professional Stance

"The drop from AUC 0.929 to 0.785 is not a step backward in performance, but a step forward in honesty." Model A is only a "dead model" that can never be deployed into a real system because, at inference time, the system has no duration data to input.

Conclusion

It is mandatory to permanently remove the duration column from the dataset. The AUC ~ 0.785 of Model B will be the most honest Baseline for the project. Subsequent chapters (Feature Engineering, SMOTE, Hyperparameter Tuning) will push this AUC higher through genuine mathematical capability — not by relying on a data "trick."

Chapter 5 Preprocessing Pipeline & Baseline Model Proposal

5.1 Preprocessing & Feature Engineering Strategy

Based on the barriers identified from raw data, the preprocessing Pipeline must be designed with purpose — absolutely not applying Scaling or Encoding functions blindly.

Remove the "Data Leakage" Tumor Critical

Model run validation showed that duration distorts Feature Importance and creates a delusional AUC. The duration column will be dropped from the very first line of code to ensure valid inference capability in a real-world environment.

Preserve Signals from "Missing Values" From Insight 1.2

Instead of using SimpleImputer to fill with Mean/Mode — which erases the truth — we will keep unknown as an independent class and encode it with One-Hot. For pdays (containing 83% value of -1), we extract a binary flag: is_previously_contacted (0/1).

Control Extreme Outliers From Insight 2.1

The balance column contains ultra-wealthy customers and bad-debt customers (negative accounts). Using MinMaxScaler or StandardScaler would collapse the distribution space. Decision: Use RobustScaler (based on IQR) for numerical variables to neutralize outlier noise points.

Dimensionality Reduction & Behavioral Clustering From Insights 2.2 & 3.4

Campaign: Bin into groups: '1 call', '2–3 calls', '4–5 calls', and '>5 calls' to learn the "Customer Fatigue" pattern.

Seasonality (Month): Group 12 months into Quarters (Q1–Q4) or use Ordinal/Target Encoding to avoid the Curse of Dimensionality.

Recommendation: Package the entire process into sklearn.compose.ColumnTransformer to ensure Train and Test sets are transformed separately, preventing Data Leakage at the Transform step.

5.2 Modeling Strategy & Performance Evaluation

Tackle Class Imbalance Definitively From Insight 1.3

The dataset is heavily skewed toward "No" (88.3%). Accuracy MUST NOT be used to measure results. We will integrate class_weight='balanced' into the algorithm, forcing heavy penalties when misclassifying the "Yes" class. The mandatory metric set: F1-Score (class 1) and ROC-AUC.

Establish Baseline Model — Logistic Regression

Deploy first to establish a benchmark (Baseline AUC ~ 0.77). Logistic Regression provides linear coefficients (Coefficients), allowing direct explanation (Explainability) to Management of which factor most strongly influences the deposit decision.

Main Weapon — Random Forest / XGBoost

Use Tree-based (Ensemble) models to push performance to the highest level. This group of algorithms can automatically handle non-linear relationships (such as the polarizing effect of job), while also recognizing cross-feature interactions between "Assets," "Education," and "Age" without requiring manually created Polynomial variables.
